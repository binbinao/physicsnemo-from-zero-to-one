## 7.0 Hook: Customers Don't Pay for Models, They Pay for Closed Loops

If you present the first 6 chapters to a customer, they'll probably nod: PINN is interesting, FNO is fast, FourCastNet is impressive.

Then they'll ask the most pragmatic question:

> **"So how does this fit into our design workflow?"**

This single question sends many AI4Science demos back to square one.

A model by itself is not a solution. What customers really want is a closed loop:

```text
Design parameters → Surrogate model → Physical fields / Performance metrics → Optimizer → Recommended design → High-fidelity validation → API service
```

Chapter 7 is that closed loop.

We use automotive aerodynamics as the main thread: the **default code** uses toy synthetic car parameters + formula $C_d$ to run through the complete loop; the **extended tier** can swap in DrivAerNet or other real CFD data. We train a surrogate model to predict $C_d$ (or pressure fields), the optimizer searches for better designs, **then validate via high-fidelity CFD** (`hifi_validation_queue.py`), and finally deploy as an API.

> **Multi-objective (C22)**: Industry commonly optimizes $C_d$, lift, pitching moment, wind noise proxy, etc. simultaneously; this book only demonstrates **single-objective $C_d$**. Multi-objective can be extended using Optuna `multi-objective study`.

This time, we're not just writing a training script. We're building a system that a solution architect can take to a demo.

DrivAerNet is a large-scale parameterized car body dataset for automotive aerodynamic design, containing numerous industry-standard DrivAer body variants with their high-fidelity CFD results. Its value lies in: **finally having a public, parameterized, near-industrial-reality automotive aerodynamic dataset**.

### 7.2.2 How Are Inputs and Outputs Defined?

This chapter starts with the most deployable objective: predicting drag coefficient $C_d$.

```text
input: car body geometric parameters / point cloud / SDF grid
output: C_d or pressure field
```

If the input is a parameterized vector:

```text
x = [body_length, body_width, body_height, front_angle, rear_angle,
     ground_clearance, wheel_diameter]   # 见 generate_toy_cars.py
y = Cd
```

If the input is a 3D geometric field:

```text
x = SDF grid [B, 1, D, H, W]
y = pressure / velocity field [B, C, D, H, W]
```

Pedagogically, we start with parameter vectors predicting $C_d$, then extend to 3D fields.

## 7.3 Custom Model: From MLP Baseline to FNO/GNO

### 7.3.1 MLP Baseline: Get the Business Loop Running First

Don't start with a 3D FNO right away. When building solutions, the first step is always a baseline.

```python
class CdMLP(torch.nn.Module):
    def __init__(self, in_dim, hidden=128, depth=4):
        super().__init__()
        layers = [torch.nn.Linear(in_dim, hidden), torch.nn.GELU()]
        for _ in range(depth - 1):
            layers += [torch.nn.Linear(hidden, hidden), torch.nn.GELU()]
        layers.append(torch.nn.Linear(hidden, 1))
        self.net = torch.nn.Sequential(*layers)

    def forward(self, params):
        return self.net(params)
```

Training:

```python
pred_cd = model(batch["params"])
loss = torch.nn.functional.mse_loss(pred_cd, batch["cd"])
```

If even this baseline can't beat simple linear regression, it means the data/features/labels have issues—don't rush to deploy a large model.

### 7.3.2 PhysicsNeMo Module Style

In the PhysicsNeMo main framework, custom models should follow `torch.nn.Module` + framework checkpoint / distributed conventions as closely as possible. Core principles:

1. `forward()` has clear inputs and outputs.
2. All configuration goes through Hydra.
3. The model can save / load state_dict.
4. Don't hardcode data preprocessing inside the model.

## 7.4 Custom PDE / Physics Regularization: When Is It Worth Adding?

The full governing equations for automotive aerodynamics are RANS / Navier-Stokes. Directly writing them as PINN loss for full-vehicle 3D CFD isn't practical. But you can still add some lightweight physics regularization:

- Divergence constraint under incompressible approximation: $\nabla\cdot u = 0$
- Far-field boundary consistency
- Pressure/velocity reasonable range constraints
- Surface pressure integral yielding $C_d$ consistent with labels

### 7.4.1 Surface Force Consistency Loss

If the model predicts a pressure field and the data also has $C_d$ labels, you can add:

$$\mathcal{L}_{force}=\left(C_d^{pred-field}-C_d^{label}\right)^2$$

This is much easier to deploy than writing full RANS residuals.

> **Engineering judgment**: The final chapter doesn't pursue "maximum physics" but rather "deliverable." Lightweight regularization is more valuable than complex but unrunnable full PDE implementations.

## 7.5 Design Optimization: From Surrogate Model to Recommended Parameters

After training $C_d=f_\theta(p)$, you can optimize design parameters $p$.

### 7.5.1 Objective Function

$$J(p)=C_d(p)+\lambda_1\text{Cost}(p)+\lambda_2\text{ConstraintViolation}(p)$$

Constraints include:

- Body height must not exceed regulatory/design limits
- Trunk space must not fall below threshold
- Styling parameters must not deviate too far from the baseline vehicle

### 7.5.2 Search with Optuna

```python
import optuna

def objective(trial):
    p = {
        "body_length": trial.suggest_float("body_length", 3.5, 5.5),
        "rear_angle": trial.suggest_float("rear_angle", 5.0, 35.0),
        "front_angle": trial.suggest_float("front_angle", 10.0, 40.0),
        # ... 其余 4 维见 optimize.py / PARAM_NAMES
    }
    x = build_feature_tensor(p).unsqueeze(0).cuda()
    with torch.no_grad():
        cd = model(x).item()

    penalty = constraint_penalty(p)
    return cd + 0.1 * penalty

study = optuna.create_study(direction="minimize")
study.optimize(objective, n_trials=500)
print(study.best_params, study.best_value)
```

### 7.5.3 Why Not Use Gradient-Based Optimization Directly?

If the input is continuous parameters, gradient optimization certainly works. But industrial design often involves discrete variables, hard constraints, and non-smooth constraints, so black-box optimizers like Optuna / BoTorch are more robust.

### 7.7.1 FastAPI Quick Version

from fastapi import FastAPI
from pydantic import BaseModel
import torch

app = FastAPI(title="PhysicsNeMo Aero Surrogate")
model = load_model("outputs/best.pt")
model.eval()

class DesignRequest(BaseModel):
    body_length: float
    body_width: float
    body_height: float
    front_angle: float
    rear_angle: float
    ground_clearance: float
    wheel_diameter: float

@app.post("/predict_cd")
def predict_cd(req: DesignRequest):
    x = build_feature_tensor(req).cuda()

    return {"cd": cd}

## 7.9 🔵 Failure Cases: 7 Pitfalls of End-to-End Systems

### Failure 1: Surrogate Model Accuracy Is High, but Optimization Results Are Invalid

The model has low MAE on the test set, but the optimizer finds bizarre designs outside the training distribution.

**Fix**: Add design space constraints; perform OOD detection on optimization results; Top-K must undergo high-fidelity validation.

### Failure 2: Only Predicting Cd, Not Looking at Pressure Fields

Accurate Cd doesn't mean the flow field is physically reasonable. Customers may need to understand why a certain design is better.

**Fix**: At minimum provide pressure field/surface pressure explanatory plots, or SHAP/sensitivity analysis.

### Failure 3: API Has No Input Validation

A customer submits an impossible body height, and the model still returns a result.

**Fix**: FastAPI schema + constraint checker + return warning.

### Failure 4: Training Data Version Is Not Traceable

Three months later the customer asks "which batch of CFD data was used for this model," and you can't answer.

**Fix**: Write data version number, Git commit, configuration, and statistics all into the checkpoint.

### Failure 5: Inference Is Fast, Preprocessing Is Slow

Model takes 10ms, CAD/SDF conversion takes 30 seconds.

**Fix**: Cache features; include the feature extraction pipeline in performance optimization.

### Failure 6: Model Drifts After Deployment

New project design space differs from training data, and errors grow.

**Fix**: Deploy OOD monitoring; route low-confidence samples back to high-fidelity simulation for incremental training.

### Failure 7: Only Selling GPUs, Not Selling Workflow

Customer thinks buying GPUs means getting results; project ultimately fails.

**Fix**: Pre-sales proposals must include data governance, baseline validation, model training, deployment, and feedback loops—not just compute pricing.

## 7.10 🔵 Uncertainty Quantification: Confidence of Surrogate Models

### Why Is UQ Needed?

Engineering sign-off requires answering not just "the model predicts $C_d=0.28$" but "what is the probability that $C_d$ exceeds 0.30?" A prediction without confidence intervals is **equivalent to no prediction** in safety-critical scenarios.

Regulatory bodies, OEMs, aviation airworthiness certification—any scenario requiring sign-off needs Uncertainty Quantification (UQ). Surrogate models inherently have an additional layer of model error compared to high-fidelity CFD; the goal of UQ is to **make this error layer visible, quantifiable, and actionable**.

### Comparison of Three Practical Methods

| Method | Principle | Implementation difficulty | Applicable scenario |
|---|---|---|---|
| **Ensemble** | Train N models with different initializations, use prediction std as uncertainty | Low (N× training cost) | General purpose, most common in industry |
| **MC-Dropout** | Keep Dropout enabled during inference, run multiple forward passes for mean and variance | Very low (change one line of code) | Quick prototyping, lightweight UQ |
| **Deep Kernel Learning / GP** | Replace the last layer with a Gaussian Process | Medium | Small data, when theoretical guarantees are needed |

### Ensemble Pseudocode Example

```python
import numpy as np

# Ensemble UQ: train N models with different seeds
models = [train_model(seed=i) for i in range(5)]

# Inference: collect predictions from all ensemble members
predictions = np.array([m(x_new) for m in models])

mean_cd = np.mean(predictions)
std_cd = np.std(predictions)  # uncertainty estimate

# 95% confidence interval (assuming Gaussian)
confidence_95 = (mean_cd - 1.96 * std_cd, mean_cd + 1.96 * std_cd)

print(f"Cd = {mean_cd:.4f} ± {1.96*std_cd:.4f} (95% CI)")
print(f"P(Cd > 0.30) ≈ {1 - norm.cdf(0.30, mean_cd, std_cd):.2%}")
```

### Integration with V&V Closed Loop

The greatest engineering value of UQ is: **it can drive an automated validation closed loop**.

When the predicted uncertainty $\sigma_{C_d}$ exceeds a set threshold, automatically add that design point to the high-fidelity validation queue:

```python
UNCERTAINTY_THRESHOLD = 0.015  # 根据业务需求设定

if std_cd > UNCERTAINTY_THRESHOLD:
    hifi_queue.append(design_point)  # 自动触发 CFD 复核
    logger.warning(f"High uncertainty ({std_cd:.4f}), queuing for CFD validation")
```

This forms an **active learning** loop: low confidence → CFD validation → new data feedback → incremental model training → confidence improvement.

### Why This Book Does Not Implement UQ

1. **Computational cost**: Ensemble requires 5× training cost, which is burdensome in a pedagogical environment.
2. **Pedagogical focus**: This book's goal is to walk you through the end-to-end basic workflow, not to cover all industrial best practices.

> ⚠️ **Strong recommendation**: In real projects, UQ **must** be implemented. A surrogate model without uncertainty estimates should not be used for any safety-related decisions.
