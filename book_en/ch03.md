## 3.0 Hook: A Heat Sink Is Not a Metal Block — It's a Profit Margin

Many people, seeing a heat sink for the first time, think it's just a piece of aluminum with a lot of fins.

CAE engineers know better.

Every fin on a heat sink represents money: taller fins dissipate heat better, but material cost, machining difficulty, and structural integrity all get worse; denser fins increase surface area, but airflow resistance rises and fan power consumption goes up; a thicker base provides a more stable thermal path, but system weight and BOM cost increase too.

So a heat sink is not a metal block. **A heat sink is a profit margin.**

When semiconductor companies do package thermal design, the real question is never "what's the temperature of this one design," but rather:

> **Given constraints on chip power, package height, fan airflow, and material cost, which combination of fin height, thickness, and spacing minimizes the peak temperature?**

The traditional approach is: modify CAD once, run Icepak once, wait once. 50 designs means waiting 50 times.

What we'll do in this chapter is: advance Chapter 2's one-dimensional heat conduction to a **heat sink case that's ready for a customer presentation**. It's still simplified, but it already exhibits the three core characteristics of industrial problems:

1. **The geometry is not a line, but a complex domain assembled from multiple solids.**
2. **There is more than one type of boundary condition — different surfaces have different physical meanings.**
3. **We don't just solve the forward problem — we also do the inverse problem: given a temperature target, infer the fin parameters.**

## 3.3 🔵 CSG Geometry: Assembling a Heat Sink from Boxes

The first barrier in industrial problems is not the PDE — it's the **geometry**.

The one-dimensional interval $[0,1]$ in Chapter 2 required no geometry modeling; a heat sink is different. It consists of a base and multiple fins, and you also want to vary fin height, thickness, and spacing for optimization.

This is where **CSG (Constructive Solid Geometry)** comes into play.

### 3.3.1 The 2D three-fin geometry used in this chapter

We first define a two-dimensional cross-section:

| Component | Dimensions | Description |
|---|---|---|
| Base | Width 1.0, height 0.2 | Chip heat enters from the bottom |
| Fin fin_1 | Width 0.08, height 0.6 | Left fin |
| Fin fin_2 | Width 0.08, height 0.6 | Middle fin |
| Fin fin_3 | Width 0.08, height 0.6 | Right fin |
| Spacing | 0.18 | Air channel between fins |

![F3.3 Three-fin 2D geometry annotated diagram](assets/f3_3_geometry.png)

### 3.3.2 Assembling geometry in code

PhysicsNeMo-Sym geometry objects support addition and subtraction operations. Pseudocode:

```python
"""ch03_heatsink/heat_sink_geometry.py"""
from physicsnemo.sym.geometry.primitives_2d import Rectangle
from physicsnemo.sym.geometry.parameterization import Parameterization, Parameter

# Parameterized variables: used later for the inverse problem
fin_h = Parameter("fin_h")
fin_w = Parameter("fin_w")

def make_three_fin_geometry(fin_height=0.6, fin_width=0.08):
    # Base: from (0,0) to (1.0,0.2)
    base = Rectangle((0.0, 0.0), (1.0, 0.2))

    # Three fins: grow upward from base top y=0.2
    fin_centers = [0.32, 0.50, 0.68]
    fins = []
    for cx in fin_centers:
        x0 = cx - fin_width / 2
        x1 = cx + fin_width / 2
        fin = Rectangle((x0, 0.2), (x1, 0.2 + fin_height))
        fins.append(fin)

    # CSG union: base + three fins
    heat_sink = base
    for fin in fins:
        heat_sink = heat_sink + fin

    return heat_sink
```

> **📌 Mental model for CSG**: You're not drawing a mesh — you're **assembling solids**. `base + fin` is a union, `domain - obstacle` is a subtraction. The geometry definition is continuous; the sampler scatters points on this continuous geometry.

### 3.3.3 Why is this better suited for PINN than meshing?

The traditional CFD/FEM workflow is:

```text
CAD → Mesh → Solver
```

The PINN workflow is more like:

```text
Parameterized geometry → Sample points → Loss function
```

The difference is: **PINN doesn't need a mesh**. It only needs to know which points are inside the geometry, which are on the boundary, and what the boundary normals are.

This is critical for design optimization. Traditional methods require re-meshing every time a fin height changes; PINN only needs to re-sample points. Once the geometry is parameterized, `fin_height` can even become a learnable variable that participates in training.

## 3.4 🔵 Governing Equation: Steady-State Conduction with Heat Source

### 3.4.1 From Chapter 2's time-varying heat conduction to steady-state conduction

In Chapter 2 we solved:

$$\frac{\partial T}{\partial t} = \alpha \nabla^2 T$$

The common engineering problem for heat sinks is the **steady-state temperature field**: the chip continuously generates heat, the heat sink continuously dissipates it, and eventually a temperature distribution is reached that no longer changes with time. Steady-state means:

$$\frac{\partial T}{\partial t} = 0$$

So the governing equation becomes:

$$\nabla \cdot (k_s \nabla T) + Q = 0$$

If the thermal conductivity $k_s$ is constant, this simplifies to:

$$k_s \nabla^2 T + Q = 0$$

Where:
- $T(x,y)$: temperature field
- $k_s$: solid thermal conductivity
- $Q$: volumetric heat source (chip heat generation)

### 3.4.2 PINN residual

The corresponding PDE residual is:

$$r_\theta(x,y) = k_s(T_{xx} + T_{yy}) + Q$$

What PINN does is drive $r_\theta$ toward 0 at interior points.

Bare PyTorch residual code:

```python
def heat_residual(model, x, y, k_s=1.0, q=0.0):
    """Steady-state conduction PDE residual: k_s * (T_xx + T_yy) + Q"""
    x = x.clone().requires_grad_(True)
    y = y.clone().requires_grad_(True)
    T = model(x, y)

    T_x = torch.autograd.grad(T, x, torch.ones_like(T), create_graph=True)[0]
    T_y = torch.autograd.grad(T, y, torch.ones_like(T), create_graph=True)[0]
    T_xx = torch.autograd.grad(T_x, x, torch.ones_like(T_x), create_graph=True)[0]
    T_yy = torch.autograd.grad(T_y, y, torch.ones_like(T_y), create_graph=True)[0]

    return k_s * (T_xx + T_yy) + q
```

In PhysicsNeMo-Sym, the PDE can be written as a symbolic class:

```python
import sympy as sp
from physicsnemo.sym.eq.pde import PDE

class HeatConduction2D(PDE):
    """Steady-state 2D heat conduction: k_s*(T_xx + T_yy) + Q = 0"""
    name = "HeatConduction2D"

    def __init__(self, k_s=1.0, q=0.0):
        x, y = sp.symbols("x y")
        T = sp.Function("T")(x, y)
        self.equations = {
            "heat_conduction": k_s * (T.diff(x, 2) + T.diff(y, 2)) + q
        }
```

> **Key change**: Chapter 2 was $u_t - \alpha u_{xx}$; Chapter 3 is $k_s(T_{xx}+T_{yy})+Q$. The essence is still second-order derivatives, just extended from one spatial dimension to two.

## 3.5 🔵 Three Types of Boundary Conditions: Dirichlet / Neumann / Robin

The real difficulty of a heat sink lies in the boundaries.

On a heat sink surface, different regions have entirely different physical meanings: the bottom is in contact with the chip, the sides may be insulated, and the fin surfaces undergo convection with air. **The same geometry may have 3–5 different boundary conditions.**

In this chapter we use the three most common boundary types:

| Boundary | Physical meaning | Mathematical expression | Usage in this chapter |
|---|---|---|---|
| Dirichlet | Directly specified temperature | $T = T_0$ | Constant temperature at bottom heat source |
| Neumann | Specified normal heat flux | $-k\nabla T\cdot n = q_n$ | Insulated side walls $q_n=0$ |
| Robin | Convective heat transfer | $-k\nabla T\cdot n = h(T-T_\infty)$ | Fin outer surfaces dissipating to air |

![F3.5 Heat sink boundary conditions annotated diagram](assets/f3_5_boundary_conditions.png)

### 3.5.1 Dirichlet: Specified temperature

The bottom is in contact with the chip, simplified as a constant heat source temperature:

$$T(x,0) = T_{hot}$$

In PhysicsNeMo-Sym this is a boundary constraint:

```python
bottom_hot = PointwiseBoundaryConstraint(
    nodes=nodes,
    geometry=geo,
    outvar={"T": 1.0},
    batch_size=cfg.batch_size.bottom,
    criteria=bottom_criteria,  # selects only the bottom face
    lambda_weighting={"T": 100.0},
)
domain.add_constraint(bottom_hot, "bottom_hot")
```

### 3.5.2 Neumann: Specified heat flux

Insulated side walls mean no heat flow crosses the boundary:

$$\nabla T \cdot n = 0$$

In PhysicsNeMo-Sym this is typically expressed via a normal derivative term, such as `normal_gradient_T` or a derivative variable generated by PDE nodes (exact name depends on version). Pedagogical pseudocode:

```python
side_insulated = PointwiseBoundaryConstraint(
    nodes=nodes,
    geometry=geo,
    outvar={"normal_gradient_T": 0.0},
    batch_size=cfg.batch_size.side,
    criteria=side_criteria,
    lambda_weighting={"normal_gradient_T": 10.0},
)
domain.add_constraint(side_insulated, "side_insulated")
```

### 3.5.3 Robin: Convective heat transfer (most important in this chapter)

A heat sink dissipates heat because its surfaces undergo convection with air. Mathematically:

$$-k_s \nabla T \cdot n = h(T - T_\infty)$$

The left side is the normal heat flux from conduction inside the solid; the right side is the heat carried away by air convection.

This type of boundary is called a **Robin boundary condition**. It's a mixture of Dirichlet and Neumann: it contains both a derivative term and the function value itself.

To write it as a PINN loss, we move everything to one side:

$$r_{robin} = -k_s \nabla T \cdot n - h(T - T_\infty) = 0$$

Pseudocode:

```python
class RobinBoundary(PDE):
    """Robin convective boundary: -k*dTdn - h*(T - T_inf) = 0"""
    def __init__(self, k_s=1.0, h=0.1, T_inf=0.0):
        x, y, normal_x, normal_y = sp.symbols("x y normal_x normal_y")
        T = sp.Function("T")(x, y)
        dTdn = T.diff(x) * normal_x + T.diff(y) * normal_y
        self.equations = {
            "robin": -k_s * dTdn - h * (T - T_inf)
        }
```

> **📌 Engineering intuition**: The $h$ in the Robin boundary is the heat transfer coefficient. The higher the fan airflow, the larger $h$, and the stronger the heat dissipation. When doing design optimization later, $h$ can represent different air-cooling configurations.

## 3.6 🔵 PhysicsNeMo-Sym's Domain / Constraint: Packaging the Physics Problem into the Framework

At this point, we have:

1. Geometry: `geo = base + fin1 + fin2 + fin3`
2. PDE: `HeatConduction2D`
3. Boundary conditions: bottom / side / fin surface

The core job of PhysicsNeMo-Sym is to assemble these three into a `Domain`.

![F3.6 Domain / Constraint structure diagram](assets/f3_6_domain_constraint.png)

### 3.6.1 Minimal structure code

```python
"""ch03_heatsink/heat_sink_train.py — pedagogical skeleton"""
import hydra
from omegaconf import DictConfig
from physicsnemo.sym.domain import Domain
from physicsnemo.sym.solver import Solver
from physicsnemo.sym.key import Key
from physicsnemo.sym.hydra import instantiate_arch
from physicsnemo.sym.domain.constraint import (
    PointwiseInteriorConstraint,
    PointwiseBoundaryConstraint,
)

from heat_sink_geometry import make_three_fin_geometry
from equations import HeatConduction2D, RobinBoundary

@hydra.main(version_base="1.3", config_path="conf", config_name="config")
def run(cfg: DictConfig):
    # 1. Geometry
    geo = make_three_fin_geometry(
        fin_height=cfg.geometry.fin_height,
        fin_width=cfg.geometry.fin_width,
    )

    # 2. Network
    net = instantiate_arch(
        input_keys=[Key("x"), Key("y")],
        output_keys=[Key("T")],
        cfg=cfg.arch.fully_connected,
    )

    # 3. Equation nodes
    heat_eq = HeatConduction2D(k_s=cfg.physics.k_s, q=cfg.physics.q)
    robin_eq = RobinBoundary(k_s=cfg.physics.k_s, h=cfg.physics.h, T_inf=cfg.physics.T_inf)
    nodes = heat_eq.make_nodes() + robin_eq.make_nodes() + [net.make_node("temperature_net")]

    # 4. Domain + Constraints
    domain = Domain()

    interior = PointwiseInteriorConstraint(
        nodes=nodes,
        geometry=geo,
        outvar={"heat_conduction": 0.0},
        batch_size=cfg.batch_size.interior,
    )
    domain.add_constraint(interior, "interior")

    bottom_hot = PointwiseBoundaryConstraint(
        nodes=nodes,
        geometry=geo,
        outvar={"T": cfg.physics.T_hot},
        batch_size=cfg.batch_size.bottom,
        criteria=bottom_criteria(),
        lambda_weighting={"T": cfg.loss_weights.bottom},
    )
    domain.add_constraint(bottom_hot, "bottom_hot")

    fin_robin = PointwiseBoundaryConstraint(
        nodes=nodes,
        geometry=geo,
        outvar={"robin": 0.0},
        batch_size=cfg.batch_size.robin,
        criteria=fin_surface_criteria(),
        lambda_weighting={"robin": cfg.loss_weights.robin},
    )
    domain.add_constraint(fin_robin, "fin_robin")

    # 5. Solver
    solver = Solver(cfg, domain)
    solver.solve()

if __name__ == "__main__":
    run()
```

> **Version note**: The exact import paths and constraint class signatures for PhysicsNeMo-Sym v2.0 may be adjusted across versions. This chapter's code shows the framework structure; the repo `ch03_heatsink/` maintains a runnable implementation compatible with the current version.

### 3.7.2 Validator: Don't trust training loss alone

One of PINN's biggest pitfalls is: **a low loss does not mean the physical solution is correct.**

This chapter's validator uses two approaches:

1. **Simplified finite difference reference**: Run a coarse finite difference solution on an approximate grid of the same 2D geometry.
2. **OpenFOAM / Icepak samples**: If you have real solver results on hand, they can be used as `PointwiseValidator` input.

Pseudocode:

```python
from physicsnemo.sym.domain.validator import PointwiseValidator

validator = PointwiseValidator(
    nodes=nodes,
    invar={"x": x_val, "y": y_val},
    true_outvar={"T": T_ref},
    batch_size=1024,
)
domain.add_validator(validator, "temperature_validator")
```

> **Engineering principle**: Any PINN result shown to a customer must have a validator. A PINN without a validator can only be called a demo, not a solution.

**This book's repo (raw PyTorch path)**: After training, `outputs/validation_report.json` is automatically generated (`ch03_heatsink/validator.py`: PDE/Robin residual + temperature range). After importing an Icepak/CFD reference field, this can be extended to a `PointwiseValidator` benchmark.

## 3.8 🔵 Inverse Problem: Inferring Fin Thickness from a Given Temperature Field

The forward problem is:

> Given geometry and materials, find the temperature field.

The inverse problem is:

> Given target or measured temperatures, infer geometry or material parameters.

The most natural inverse problem for a heat sink is: **If we want the peak temperature below a certain threshold, how tall, thick, and spaced should the fins be?**

### 3.8.1 Why is the inverse problem painful for traditional solvers?

Traditional Icepak / Fluent / OpenFOAM can solve forward problems very well, but inverse problems typically require wrapping an optimizer around the outside:

```text
Candidate geometry parameters → Rebuild CAD → Re-mesh → Solve → Compute objective → Optimizer proposes next parameters
```

One cycle might take 1 hour; 100 cycles means 100 hours.

PINN's approach is different: if geometry parameters are in the differentiable computation graph, **the geometry parameters themselves can be optimized**.

### 3.8.2 Parameterizing fin height

First define the parameter:

```python
from physicsnemo.sym.geometry.parameterization import Parameterization, Parameter

fin_h = Parameter("fin_h")
param_ranges = {fin_h: (0.4, 0.8)}
parameterization = Parameterization(param_ranges)
```

Then the geometry uses `fin_h` instead of a fixed value:

```python
fin = Rectangle((x0, 0.2), (x1, 0.2 + fin_h), parameterization=parameterization)
```

During training, the network input is no longer just $(x,y)$ but also includes geometry parameters:

```python
net = instantiate_arch(
    input_keys=[Key("x"), Key("y"), Key("fin_h")],
    output_keys=[Key("T")],
    cfg=cfg.arch.fully_connected,
)
```

This means the network learns:

$$T = f_\theta(x, y, h_{fin})$$

Not the temperature field of a single geometry, but the temperature field of a family of geometries.

### 3.8.3 Optimization objective: minimize peak temperature

We want the peak temperature near the chip to be as low as possible, while not letting the fins grow infinitely tall (otherwise the answer is always "taller is better"). Define the objective:

$$J(h_{fin}) = \max_{(x,y) \in \Omega_{chip}} T_\theta(x,y,h_{fin}) + \lambda \cdot h_{fin}$$

The first term is the peak temperature; the second is a material cost penalty.

Optimization procedure:

```python
# After training the parameterized PINN, freeze network weights and only optimize fin_h
fin_h_var = torch.tensor([0.5], requires_grad=True)
optimizer = torch.optim.Adam([fin_h_var], lr=1e-2)

for step in range(500):
    optimizer.zero_grad()
    T_chip = model(x_chip, y_chip, fin_h_var.expand_as(x_chip))
    objective = T_chip.max() + 0.05 * fin_h_var
    objective.backward()
    optimizer.step()
    fin_h_var.data.clamp_(0.4, 0.8)
```

![F3.8 Inverse problem optimization flowchart](assets/f3_8_inverse_flow.png)

### 3.8.4 How to interpret the results?

A reasonable result looks like this:

| Design | fin_height | Peak temperature | Material cost proxy | Combined objective |
|---|---:|---:|---:|---:|
| Initial design | 0.50 | 1.000 | 0.050 | 1.050 |
| Heat dissipation only | 0.80 | 0.742 | 0.080 | 0.822 |
| With cost penalty | 0.68 | 0.781 | 0.068 | **0.849** |

The key here is not the specific numbers, but the workflow: **train a surrogate model once, then perform second-scale optimization on the surrogate**.

## 3.9 🏭 Industry Mapping: Chip Package Thermal Design Closed Loop

Now let's place this chapter's toy example back into the real semiconductor workflow.

### 3.9.1 Traditional workflow

```text
Customer provides power / package constraints
    ↓
Thermal design engineer builds CAD
    ↓
Icepak / Flotherm modeling + meshing
    ↓
Solve: 1–4 hours
    ↓
Check if peak temperature exceeds spec
    ↓
Modify fins / materials / fan
    ↓
Repeat 30–100 times
```

### 3.9.2 PhysicsNeMo surrogate model workflow

```text
First generate 50–200 high-quality cases with traditional solvers
    ↓
Train parameterized PINN / neural operator surrogate model
    ↓
New designs: second-scale inference
    ↓
Run parameter optimization on the surrogate model
    ↓
Send only the best 3–5 designs back to Icepak for high-fidelity verification
```

## 3.10 🔵 Failure Cases: Geometry, Boundary, and Sampling Are Where Things Go Wrong Most Easily

Starting from Chapter 3, failure is no longer just "loss won't decrease." Geometry and boundaries introduce new pitfalls.

### Failure 1: Geometry assembled wrong — sample points fall in the air

**Symptom**: The temperature field has predictions outside the fins, or the visualization contour is incorrect.

**Cause**: CSG union / subtract written incorrectly, or coordinate ranges are inconsistent.

**Debugging**: Before training, plot `geo.sample_interior()` and `geo.sample_boundary()` as scatter plots. **No sampling plot, no training.**

### Failure 2: Boundary criteria selects the wrong region

**Symptom**: The bottom constant-temperature condition is not taking effect, or the top convection is applied to the bottom.

**Cause**: `criteria` is too broad — e.g., `y < 0.21` accidentally includes the top of the base.

**Debugging**: Sample each constraint individually and plot with color-coded bottom/side/robin.

### Failure 3: Robin loss stuck and won't decrease

**Symptom**: interior/bottom both decrease, but `robin` is stuck at 1e-1.

**Possible causes**:
1. Normal vector sign is flipped.
2. $h$ magnitude is too large, making the boundary term stiff.
3. `lambda_weighting` is too small.

**Fix**: First reduce $h$ by 10× for debugging; confirm the normal vector direction; then adjust weights.

### Failure 4: Non-physical oscillations in the temperature field

**Symptom**: Isothermal lines show ripples at the fin root.

**Cause**: Gradient discontinuity at sharp geometry corners makes it hard for PINN to fit.

**Fix**:
- Use importance sampling near the fin root.
- Increase network width.
- Use Fourier features / positional encoding.

### Failure 5: Inverse problem gives boundary-value answers

**Symptom**: Optimization result is always `fin_h=0.8` (the maximum value).

**Cause**: The objective function only penalizes peak temperature with no cost term — of course taller is always better.

**Fix**: Add material cost / flow resistance / package height constraints, or change to a multi-objective Pareto formulation.

### 5-Minute Self-Check List

- [ ] Did you plot geometry interior / boundary sampling?
- [ ] Did you visualize each boundary constraint individually?
- [ ] Did you confirm the Robin normal vector direction?
- [ ] Are the loss weights causing all four losses to decrease together?
- [ ] Does a validator exist?
- [ ] Does the inverse problem objective include cost/constraints?
